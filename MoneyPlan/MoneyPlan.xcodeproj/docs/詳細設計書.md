# MoneyPlan 詳細設計書

## 1. 文書概要
- 文書名: MoneyPlan 詳細設計書
- 作成日: 2026-03-11
- 対象: iOS アプリ `MoneyPlan`
- 参照元:
  - `要件定義書.md`
  - `基本設計書.md`

本書は、基本設計書で定義した構成を実装可能な粒度まで分解した詳細設計を示す。
画面ごとの責務、状態、イベント、データ構造、サービス責務、保存処理、再計算処理、テスト観点を定義し、実装時の判断ぶれを抑えることを目的とする。

## 2. 実装前提
### 2.1 対応範囲
- 初期版は iPhone 縦画面を主対象とする
- 対応 OS は iOS 18 以上とする
- データは端末内保存のみとし、同期や共有は扱わない
- 通貨は日本円固定とする
- 管理口座は 1 口座固定とする

### 2.2 採用技術
- UI: SwiftUI
- 永続化: SwiftData
- グラフ: Charts
- テスト: XCTest / XCUITest

### 2.3 実装ディレクトリ方針

## 3. アーキテクチャ詳細
### 3.1 層構成
#### Presentation 層
- 画面描画
- ユーザー入力受付
- 画面状態管理
- 保存や削除のトリガ

#### Application / Domain 層
- 残高計算
- 月次集計
- 定期予定の展開
- 警告判定
- 表示用データ組み立て

#### Data 層
- SwiftData モデル定義
- フェッチ条件の集約
- 保存、更新、削除

### 3.2 責務分担
| 単位 | 責務 |
| --- | --- |
| `View` | UI 描画、画面イベントの発火 |
| `ViewModel` | 入力状態、バリデーション、保存要求の組み立て |
| `Repository` | SwiftData へのアクセス集約 |
| `Service` | 業務ロジック、計算、集計 |
| `Model` | 永続化対象のデータ表現 |

### 3.3 状態管理方針
- 一覧系画面は `@Query` または取得結果を `ViewModel` に投影して描画する
- フォーム系画面は `Observable` な `ViewModel` に入力値を保持する
- 保存完了後は SwiftData 更新を起点に一覧と集計を再評価する
- 画面固有の一時状態は `ViewModel`、永続状態は `Model` で分離する

### 4.5 SCR-03 予定編集画面
#### 入力項目
| 項目 | 型 | 必須 | 備考 |
| --- | --- | --- | --- |
| `selectedRecurringPlanID` | `UUID?` | No | 選択時は定期予定から初期値反映 |
| `flowType` | `FlowType` | Yes | 入金 / 出金 |
| `date` | `Date` | Yes | 日付のみ利用 |
| `name` | `String` | Yes | 前後空白を除去して保存 |
| `amountText` | `String` | Yes | 画面入力値 |
| `memo` | `String` | No | 改行可 |
| `sortOrderText` | `String` | Yes | 未入力時は `0` 扱い |

#### 表示モード
- `create`: 新規登録
- `edit`: 既存予定編集

#### 定期予定選択時の挙動
1. 定期予定を未選択なら通常の自由入力とする
2. 定期予定を選択した場合は `name`、`amountText`、`flowType` の初期値へ反映する
3. 反映後の入力値はユーザーが個別に変更できる
4. 定期予定選択は補助入力であり、保存時に定期予定由来予定へ強制変換しない

#### バリデーション
| 項目 | 条件 | エラー文言 |
| --- | --- | --- |
| `name` | 空文字不可 | `名称を入力してください` |
| `amountText` | 数値変換可かつ 1 以上 | `金額は 1 円以上で入力してください` |
| `sortOrderText` | 数値変換可かつ 0 以上 | `表示順は 0 以上で入力してください` |

#### 保存処理
1. 入力値を trim する
2. バリデーション実行
3. `PlanEditorInput` を生成する
4. `PlanRepository` に保存依頼
5. 保存対象が自動生成予定かつ主要項目変更時は `isOccurrenceLocked = true` を付与する
6. 保存後にシートを閉じる

#### 削除処理
1. 編集モード時のみ削除ボタン表示
2. 確認ダイアログ表示
3. 削除実行
4. 関連集計を更新
5. シートを閉じる

## 5. モデル詳細設計
### 5.1 `FlowType`
```swift
enum FlowType: String, Codable {
    case income
    case expense
}
```

### 5.2 `TransactionPlan`
```swift
@Model
final class TransactionPlan {
    @Attribute(.unique) var id: UUID
    var date: Date
    var flowTypeRawValue: String
    var name: String
    var amount: Int
    var memo: String
    var sortOrder: Int
    var sourceTemplateID: UUID?
    var isAutoGenerated: Bool
    var isOccurrenceLocked: Bool
    var createdAt: Date
    var updatedAt: Date
}
```

#### 制約
- `amount` は正数のみ保持する
- `memo` は未入力時に空文字で保持する
- `sortOrder` は 0 以上とする

#### 派生値
- `flowType`: `flowTypeRawValue` から導出
- `signedAmount`: `income` は正、`expense` は負

### 5.3 `RecurringPlan`
```swift
@Model
final class RecurringPlan {
    @Attribute(.unique) var id: UUID
    var flowTypeRawValue: String
    var name: String
    var amount: Int
    var dayOfMonth: Int
    var startMonth: Date
    var endMonth: Date?
    var isActive: Bool
    var note: String
    var createdAt: Date
    var updatedAt: Date
}
```

#### 制約
- `dayOfMonth` は 1 から 28
- `amount` は正数
- `note` は未入力時に空文字で保持する

### 5.4 `AppSetting`
```swift
@Model
final class AppSetting {
    @Attribute(.unique) var id: UUID
    var initialBalance: Int
    var warningBalanceThreshold: Int
    var createdAt: Date
    var updatedAt: Date
}
```

#### 制約
- `AppSetting` は常に 1 レコードのみを維持する
- 初回起動時に存在しなければ自動生成する
- 複数レコードが存在する場合は最も古い 1 件を採用し、それ以外は整合性回復対象とする

## 8. Service 詳細設計
### 8.1 `BalanceCalculator`
#### 入力
- `initialBalance`
- `plans`
- `warningThreshold`

#### 出力
- `BalanceCalculationResult`

#### 出力項目
| 項目 | 型 | 内容 |
| --- | --- | --- |
| `entries` | `[BalanceEntry]` | 各予定時点の残高 |
| `currentBalance` | `Int` | 最新残高 |
| `lowestBalance` | `Int` | 最低残高 |
| `firstNegativeDate` | `Date?` | 初回残高不足日 |
| `warningDates` | `[Date]` | 閾値警告日 |

#### 処理手順
1. 予定をソートする
2. `initialBalance` を起点値にする
3. 予定ごとに `signedAmount` を加算する
4. `BalanceEntry` を順に生成する
5. 最低残高と警告判定を更新する

### 8.2 `RecurringPlanGenerator`
#### 責務
- 定期予定から将来 `TransactionPlan` を生成する
- 既存の自動生成予定を更新または削除する
- 個別ロック済み予定を保護する

#### 入力
- `templates`
- `existingGeneratedPlans`
- `generationStartMonth`
- `generationMonthCount`

#### 生成ルール
1. 対象月列を月初基準で作成する
2. 各テンプレートごとに適用対象月だけ処理する
3. 発生日は `dayOfMonth` をそのまま採用する
4. 同一テンプレート、同一月の自動生成予定が未存在なら作成する
5. 既存自動生成予定があり `isOccurrenceLocked = false` なら上書きする
6. テンプレート停止または終了範囲外になった未来予定は削除する

#### 非対象
- 29 日以降の月末補正
- 祝日繰上げ、繰下げ
- 複雑な繰り返し周期

### 8.3 `DashboardSummaryBuilder`
#### 責務
- ダッシュボード表示に必要な集計を 1 つのモデルへまとめる

#### 処理
1. 全予定の残高計算結果を受け取る
2. 当月範囲の入出金合計を算出する
3. 当日以降の直近 5 件を抽出する
4. `graphRange` に応じて `直近 1 か月` または `全期間` のグラフ点を抽出する
5. 警告状態を決定する

### 8.4 `MonthlySummaryBuilder`
#### 責務
- 月次画面用の日別集計とサマリー算出

#### 処理
1. 対象月の予定を抽出する
2. 日別に入金合計、出金合計、件数を集計する
3. 当月末残高と最低残高を算出する
4. `MonthlySummary` を返却する

## 9. フロー詳細設計
### 9.1 アプリ初回起動
1. `SettingRepository.fetchOrCreate()` を実行する
2. 必要に応じて `normalizeIfNeeded()` で設定重複を解消する
3. `RecurringPlanRepository.fetchActivePlans()` を取得する
4. `RecurringPlanGenerator` で当月から 12 か月分を生成する
5. 画面表示用の集計を構築する

### 9.2 予定新規登録
1. ユーザーが入力
2. `PlanEditorViewModel` が検証
3. `PlanRepository.save()` を実行
4. SwiftData 保存
5. 一覧、ダッシュボード、月次サマリーが再取得される

### 9.3 予定編集
1. 編集対象をフォームへ反映
2. ユーザー編集後に検証
3. 保存時に自動生成予定ならロック要否を判定
4. 更新保存
5. 画面を閉じて再描画

### 9.4 定期予定更新
1. テンプレート保存
2. `RecurringPlanGenerator` 実行
3. 将来自動生成予定を追加、更新、削除
4. 残高と集計を再描画

### 9.5 設定変更
1. 設定値検証
2. `AppSetting` 保存
3. `BalanceCalculator` を再実行
4. 警告表示とグラフを更新

## 10. バリデーション・エラー処理詳細
### 10.1 共通方針
- 画面遷移前ではなく保存操作時に検証する
- エラーはフォーム上部に要約、項目付近に個別表示する
- 保存失敗時は内容を維持したまま再試行可能にする

### 10.2 想定エラー
| 区分 | 発生箇所 | 対応 |
| --- | --- | --- |
| 入力不正 | 各編集画面 | バリデーション文言表示 |
| 保存失敗 | Repository | アラート表示、再試行可能 |
| 設定欠損 | 起動時 | 初期設定を自動生成 |
| データ不整合 | `flowTypeRawValue` 不正など | 不正データを除外しログ出力 |

### 10.3 削除確認文言
- 予定削除: `この予定を削除します。`
- 定期予定削除: `この定期予定を削除し、未ロックの将来予定も整理します。`
