# MoneyPlan 基本設計書

## 1. 文書概要
- 文書名: MoneyPlan 基本設計書
- 作成日: 2026-03-11
- 対象: iOS アプリ `MoneyPlan`
- 参照元: `要件定義書.md`

本書は、要件定義書で定義した MVP を iPhone アプリとして実装するための基本設計を示す。
画面構成、アーキテクチャ、データ設計、主要処理、検証方針を定義し、詳細設計と実装の基準とする。

## 2. 設計方針
### 2.1 基本方針
- 初期版は 1 ユーザー、1 口座、1 通貨、1 端末内保存を前提とする
- 元の `Book1.numbers` で行っていた「日付順の資金繰り確認」と「将来残高の把握」を最優先とする
- UI は SwiftUI、データ永続化は SwiftData、グラフは Charts を用いる
- 予定と残高計算の整合性を最優先とし、一覧、ダッシュボード、グラフで同一ロジックを利用する
- 将来の CSV 入出力、通知、複数口座対応を見据え、画面ロジックと計算ロジックを分離する

### 2.2 採用技術
- 対応 OS: iOS 18 以上
- UI: SwiftUI
- 永続化: SwiftData
- グラフ: Charts
- テスト: XCTest

## 3. システム構成
### 3.1 構成概要
アプリは以下の 3 層で構成する。

1. Presentation 層
   画面表示、入力、画面遷移、ユーザー操作の受付を担う。
2. Application / Domain 層
   残高計算、月次集計、定期予定展開、警告判定などの業務ロジックを担う。
3. Data 層
   SwiftData モデル定義と保存、取得、更新を担う。

## 6. 主要処理設計
### 6.1 残高計算
#### 並び順
予定は以下の優先順でソートして計算する。

1. `date` 昇順
2. `sortOrder` 昇順
3. `createdAt` 昇順
4. `id` 昇順

#### 計算式
- `signedAmount = flowType == income ? amount : -amount`
- `runningBalance = initialBalance + 累積 signedAmount`
- `monthlyIncomeTotal = 対象月の income 合計`
- `monthlyExpenseTotal = 対象月の expense 合計`
- `projectedLowestBalance = 対象範囲の runningBalance 最小値`

#### 判定
- `runningBalance < 0` の最初の日を残高不足日とする
- `runningBalance <= warningBalanceThreshold` の日を閾値警告対象とする

### 6.2 定期予定の生成
#### 生成タイミング
- アプリ初回起動時
- `RecurringPlan` の追加時
- `RecurringPlan` の編集時
- `RecurringPlan` の停止 / 再開時
- 月切替時に生成範囲不足を検知した場合

#### 生成範囲
- 当月から 12 か月先までを生成対象とする
- 生成範囲は定数化し、将来変更しやすくする

#### 生成ルール
1. 対象 `RecurringPlan` が `isActive = true` である
2. 対象月が `startMonth` 以降かつ `endMonth` 以前である
3. 対象日の `TransactionPlan` が未生成である
4. 既存自動生成予定があり `isOccurrenceLocked = false` の場合は再生成で置き換える
5. `isOccurrenceLocked = true` の場合は保持する

### 6.3 予定 CRUD
#### 登録
1. 入力値を検証
2. `TransactionPlan` を保存
3. 該当日以降の残高を再計算
4. 一覧、ダッシュボード、グラフを再描画

#### 編集
1. 対象データを読み込み
2. 入力値を検証
3. 更新保存
4. 必要に応じて `isOccurrenceLocked = true` を付与
5. 変更日以降の残高を再計算

#### 削除
1. 削除確認を表示
2. 対象データを削除
3. 削除日以降の残高を再計算

### 6.4 設定変更
1. `initialBalance` または `warningBalanceThreshold` を保存
2. 全期間の残高と警告判定を再計算
3. ダッシュボードと一覧へ即時反映

## 8. 入力バリデーション
### 8.1 予定
- `name` は 1 文字以上
- `amount` は 1 円以上
- `date` は必須
- `sortOrder` は 0 以上

### 8.2 定期予定
- `name` は 1 文字以上
- `amount` は 1 円以上
- `dayOfMonth` は 1 から 28
- `endMonth` 指定時は `startMonth <= endMonth`

### 8.3 設定
- `initialBalance` は 0 円以上を推奨値とするが、負数も保存可能とする
- `warningBalanceThreshold` は 0 円以上

## 9. エラー処理設計
- 保存失敗時はアラートを表示し、未保存であることを明示する
- 入力不正時は項目下またはフォーム上部にエラーを表示する
- 残高再計算失敗時は一覧更新を中断し、再試行導線を表示する
- データ破損時に備え、初回ロード時は `AppSetting` 不在なら初期レコードを自動生成する

## 10. テスト方針
### 10.1 単体テスト
- `BalanceCalculator`
  - 入金 / 出金が混在する場合の累計残高
  - 同一日の `sortOrder` 反映
  - 初期残高変更時の再計算
  - 残高不足日、閾値警告日の判定
- `RecurringPlanGenerator`
  - 開始月、終了月、有効状態の判定
  - 更新時の再生成
  - `isOccurrenceLocked` の保持
- `MonthlySummaryBuilder`
  - 月次収支合計
  - 最低残高算出

### 10.2 UI テスト
- 予定の新規追加
- 予定の編集、削除
- 定期予定の追加と停止
- 設定変更後のダッシュボード反映

## 11. 今回の設計判断
- 未確定事項だった口座数は、初期版では単一口座前提で設計する
- 毎月発生日は MVP では 28 日まで対応とし、月末補正ロジックは後続対応とする
- 定期予定は当月から 12 か月先までのローリング生成で実装する
- 月次サマリーは独立タブではなく補助画面として配置する
