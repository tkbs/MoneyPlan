## 3. 実装方針
- 設計書の推奨順を維持しつつ、画面単位ではなく「保存」「計算」「表示」が揃う単位で進める。
- 既存の `Repository` と `Service` を再利用し、画面側へ業務ロジックを持ち込まない。
- 一覧とダッシュボードと月次サマリーで残高計算ロジックを共有する。
- UI 実装ごとに最低限のビルド確認を行い、計算ロジック追加時は単体テストを先行する。

- ロジック追加時:

- 画面追加時:

  - Simulator 上で `xcodebuild test` を実行する

## 7. リスクと注意点
- `TransactionPlan` の残高表示は全期間計算を前提にしているため、一覧用の月次絞り込みで計算ロジックを重複させない。
- 定期予定の更新と自動生成予定の整合性は、`isOccurrenceLocked` の扱いを崩すと破綻しやすい。
- `AppSetting` は 1 レコード前提なので、将来機能追加時も複数設定を前提にしない。
- 画面実装を急ぎすぎると `View` 側に保存処理が漏れやすいため、`ViewModel` と `Repository` の責務分離を維持する。
